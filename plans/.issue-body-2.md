## What to build

Pełen end-to-end flow podpięcia i odpięcia banku, bez sync transakcji (sync to Phase 3). Owner widzi w ustawieniach gospodarstwa nową sekcję "Bank-sync" z dwoma akcjami: lista podpiętych banków (na początku pusta) + przycisk "Połącz bank".

Po kliknięciu "Połącz bank":

1. Apka pyta GC o listę polskich banków → renderuje grid kafelków z logami z GC.
2. Owner wybiera np. PKO → redirect na hosted page GC → bank login + zgoda PSD2.
3. Callback wraca do apki → owner widzi listę swoich rachunków (osobiste, oszczędnościowe, walutowe) z checkboxami → zaznacza tylko te które chce sync'ować → zatwierdza.

Lista podpiętych banków pokazuje dla każdego: instytucję, status (active / expiring / expired / revoked), liczbę aktywnych kont, datę wygaśnięcia zgody, przycisk "Odłącz" (woła GC delete requisition + kasuje powiązania w naszej DB).

Tabele `bank_connections` i `bank_accounts` powstają w tej fazie (migracja drizzle). Wszystkie endpointy i UI są owner-only — member nie widzi sekcji "Bank-sync" w ogóle. Endpoint `POST /api/bank/connect` używa `assertGcCapacity()` z Phase 1 i zwraca 503 z `gc_limit_reached` gdy konto GC pełne.

Szczegóły schematu danych i flow w sekcji "Architectural decisions" i Phase 2 planu.

## Acceptance criteria

- [ ] Migracja drizzle dodaje `bank_connections` + `bank_accounts`
- [ ] Klient GC umie: `listInstitutions(country)`, `createRequisition({...})`, `getRequisition(id)`, `getAccountDetails(id)`, `deleteRequisition(id)` — każdy z testem na mock fetch
- [ ] Endpointy: `POST /api/bank/connect`, `GET /api/bank/callback`, `POST /api/bank/connections/:id/accounts`, `GET /api/bank/connections`, `DELETE /api/bank/connections/:id`
- [ ] Wszystkie endpointy owner-only (403 dla member, integration test)
- [ ] Frontend: grid banków z logami, redirect na hosted page, ekran post-callback z wyborem kont, lista podpiętych banków
- [ ] Sekcja "Bank-sync" w ustawieniach gospodarstwa widoczna tylko ownerowi
- [ ] Manual E2E: prawdziwe podpięcie testowego konta przez GC sandbox + odłączenie

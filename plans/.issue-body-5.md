## Type

**HITL** — acceptance wymaga manualnego ustawienia `expires_at` w DB do symulacji + sprawdzenia wysłanego emaila.

## What to build

Obsługa cyklu 90 dni zgody PSD2 żeby sync nie znikał po cichu. Trzy elementy: cron renewal-checker, emaile przez Resend, banner w UI z flow odnowienia.

**(1) Cron renewal-checker.** Codzienny Cloudflare Cron Trigger (oddzielny od 4h sync) sprawdza `bank_connections WHERE expires_at - NOW() <= 7 days`. Dla każdej takiej connection — jeśli `last_warned_at IS NULL OR < 6 dni temu` → wysyła email i aktualizuje `last_warned_at`. Daje to dwie sensowne notyfikacje: pierwsza około 7 dni przed, druga około 1 dzień przed.

**(2) Email template PL przez Resend.** Krótka wiadomość: "Cześć, Twoja zgoda na sync z bankiem PKO (lub innym) wygasa za X dni. Żeby sync nie stanął, kliknij ten link i odnów zgodę: [LINK do apki na dashboardzie]". Adresat = `users.email` ownera gospodarstwa do którego należy connection.

**(3) UI: RenewConsentBanner.** Nowy komponent w dashboardzie, widoczny tylko ownerowi gdy w jego gospodarstwie istnieje co najmniej jedna connection w stanie expiring (<=7 dni do expiry) lub expired. Żółty kolor dla expiring, czerwony dla expired. Treść: "Twoja zgoda dla [bank name] wygasa za X dni — odnów żeby sync nie stanął" + przycisk "Odnów".

**(4) Flow renewal.** Klik "Odnów" prowadzi przez ten sam flow co Phase 2 (#16) — grid banków z preselected institution → bank login → wybór kont. Backend rozpoznaje to jako renewal po parametrze `renew_connection_id` → zastępuje stary `bank_connections` row in-place (zachowuje `id` żeby `transaction_inbox.bank_connection_id` nie zostały sierotami). Jeśli te same `gc_account_id` zwrócone — `bank_accounts` zachowane, inaczej user wybiera nowe.

## Acceptance criteria

- [ ] Codzienny Cloudflare Cron Trigger zarejestrowany (oddzielny route od 4h sync)
- [ ] Renewal-checker iteruje `bank_connections WHERE expires_at - NOW() <= 7 days` i wysyła email przez Resend gdy `last_warned_at IS NULL OR < 6 days ago`
- [ ] Email template PL z linkiem do dashboardu — testowalny przez mock Resend
- [ ] `last_warned_at` aktualizowany po wysłaniu — drugi email dopiero >=6 dni od pierwszego
- [ ] `RenewConsentBanner` na dashboardzie, owner-only, żółty (expiring) / czerwony (expired)
- [ ] Klik "Odnów" → flow z Phase 2 z parametrem `renew_connection_id` → backend zastępuje row in-place (zachowuje `id`)
- [ ] Jeśli te same `gc_account_id` po renewal → `bank_accounts` zachowane bez zmian. Inaczej user wybiera ponownie z `bank_accounts.enabled` zsynchronizowanym z wyborem.
- [ ] Integration test: insert connection z `expires_at = NOW + 7d` → odpal cron → email log + status banneru
- [ ] Manual E2E: ustawić `expires_at` w DB ręcznie na +1d → odebrać email → przejść flow renew → `connection.expires_at` się odnawia, `transaction_inbox` nadal działa

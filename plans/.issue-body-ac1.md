## What to build

Najcieńsza kompletna ścieżka end-to-end termostatu, dowodząca najbardziej ryzykownej integracji (cron → decyzja → realna komenda Tuya), z pogodą zastąpioną wartością wpisywaną do konfiguracji.

- Migracja tabeli `ac_thermostats` (pełny kształt z PRD: `enabled`, `location_label`, `lat`, `lon`, `temp_on`, `temp_off`, `last_action`, `last_outdoor_temp`, `last_checked_at`; 1:1 do `smart_devices`, powiązanie z `household`).
- API do zapisu/odczytu konfiguracji termostatu per `ir_ac` (na tym etapie temperatura testowa wpisywana wprost zamiast lat/lon + API pogody).
- Czysta funkcja `decide({ temp, tempOn, tempOff, lastAction })` z pełną histerezą + edge-triggerem.
- Gałąź crona `minuta % 30 === 0` w handlerze `scheduled`: pobranie aktywnych termostatów (JOIN do `smart_devices` + `tuya_credentials`, cache tokenu Tuya per gospodarstwo), `decide()`, ewentualny `sendAcCommand(power)`, zapis `last_action`.

Szczegóły w PRD #48 (sekcje „Implementation Decisions" i „Validation Strategy") oraz w `plans/ac-thermostat.md` (Faza 1).

## Acceptance criteria

- [ ] Testy jednostkowe `decide()` przechodzą dla pełnej tablicy przypadków (poniżej/powyżej/strefa martwa/duplikat akcji/granice ≥ i ≤/`lastAction=null`).
- [ ] Zapis konfiguracji termostatu dla urządzenia `ir_ac` działa i jest powiązany z gospodarstwem.
- [ ] Gałąź crona `%30` odpala runner; pozostałe gałęzie (timery 5 min, energia 15 min, ST 12h) działają bez zmian.
- [ ] Test integracyjny (wzorzec projektu, realny Tuya test): testowa temp ≥ `temp_on` i `last_action != 'on'` → wysłano `power=1`, zapisano `last_action='on'`; analogicznie wyłączenie; strefa martwa → brak komendy.
- [ ] Wpis z `last_action='on'` przy temp nadal powyżej progu nie wysyła powtórnej komendy.

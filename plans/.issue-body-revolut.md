## Kontekst i relacja do PRD #14

To **wąski, samodzielny feature** dla integracji z Revolut Business API, niezależny od głównego planu bank-sync (PRD #14) opartego o GoCardless. Powstaje dlatego, że GoCardless ma w 2026 zamknięte signupy dla nowych developerów, a polskie banki przez agregator stały się niedostępne. Revolut Business API jest jedyną realną alternatywą która nie wymaga TPP license z KNF.

**Świadome ograniczenie zakresu**: tylko **admin apki** (właściciel Revolut Business account) podpina swoje konto. Pozostali owner-zy gospodarstw nie korzystają z tego feature'a — dla nich pozostaje manualne wpisywanie wydatków do czasu otwarcia signupów GoCardless. Ten feature jest "lepsze niż nic dla Piotra".

## Problem statement

Jako admin apki chcę żeby transakcje z mojego konta Revolut Business wpadały automatycznie do mojego gospodarstwa, zamiast wpisywać je ręcznie. To eliminuje znaczną część manualnej pracy dla mnie, nie ruszając UX innych gospodarstw.

## Solution

Cienka integracja z Revolut Business API. Admin podpina swoje konto Revolut Business przez generowanie API tokena w panelu Revolut i wkleja klucz do Cloudflare Workers secrets. Cron co 4h pobiera nowe transakcje, filtruje, mapuje na inbox z badge `🅁 Revolut`. Reszta UX (inbox, kategoryzacja, dedupe, edit-guard) — wzorowana na architekturze z PRD #14 ale **wbudowana minimalnie w to issue** zamiast czekać na merge issues #15-20 (które stoją zamrożone).

Jeśli/gdy GC otworzy signupy i wdrożymy PRD #14, oba feature'y współistnieją: użytkownicy z Revolut Business używają tej integracji, użytkownicy z PKO/ING korzystają z GC. Inbox i reguły kategoryzacji wspólne dla obu źródeł.

## User stories

1. Jako admin apki chcę wejść w ustawienia gospodarstwa i wkleić swój Revolut Business API token (lub zobaczyć w README jak go wygenerować), żeby aktywować sync z Revolut.
2. Jako admin chcę widzieć status swojego podpięcia (active / token wygasł / błąd) z linkiem "Wygeneruj nowy token" do panelu Revolut.
3. Jako admin chcę żeby co 4h apka sama pobierała nowe transakcje z mojego Revolut Business, bez mojej akcji.
4. Jako admin chcę móc kliknąć "Sync teraz" żeby wymusić natychmiastowy pobór.
5. Jako admin chcę widzieć nowe transakcje w **Inboxie** z dropdown kategorii i przyciskami Zatwierdź/Pomiń przed wpadnięciem do budżetu.
6. Jako admin chcę móc zaznaczyć "zapamiętaj regułę: opis zawiera X → kategoria Y" przy zatwierdzaniu, żeby kolejne podobne transakcje były auto-kategoryzowane.
7. Jako admin chcę żeby zatwierdzone wpisy pokazywały badge `🅁 Revolut` w listach przychodów/wydatków, żebym odróżniał od ręcznych.
8. Jako admin chcę żeby PATCH na takim wpisie blokował zmianę kwoty i daty (bank to source of truth), ale pozwalał zmienić kategorię i nazwę.
9. Jako admin chcę widzieć w historii zmian jeden bulk wpis "Sync z Revolut: X transakcji" zamiast spamu, i móc cofnąć cały batch w okienku 24h.
10. Jako admin chcę żeby przelewy między moimi własnymi rachunkami Revolut nie pojawiały się jako wydatki (auto-skip po typie).
11. Jako admin chcę dostać email gdy sync zacznie się sypać (np. token wygasł albo API zwraca błędy 3x z rzędu), żebym wiedział że trzeba odnowić.
12. Jako member gospodarstwa NIE widzę żadnych UI związanych z Revolut sync — to tylko sprawa admina.

## Implementation Decisions

### Architektura

- **Provider boundary**: Revolut Business REST API (`https://b2b.revolut.com/api/1.0/`). Token JWT-based, generowany w panelu Revolut Business, ważny ~40 minut access + refresh_token na odnawianie.
- **Storage tokenów**: `REVOLUT_REFRESH_TOKEN` w Cloudflare Workers secrets. Access token cache'owany w pamięci Workera (z TTL) — refreshowany on-demand przed expirycją.
- **Pojedynczy admin**: w przeciwieństwie do GC, nie ma multi-tenant. Tylko jeden Revolut token globalnie. Email admina z `ADMIN_EMAILS` (Phase 1 #15) decyduje kto jest "Revolut admin".
- **Gospodarstwo docelowe**: transakcje wpadają do gospodarstwa którego owner-em jest admin apki. Jeśli admin jest member-em więcej niż jednego gospodarstwa — wybiera w settings które gospodarstwo dostaje sync (singleton wybór, w env `REVOLUT_SYNC_HOUSEHOLD_ID` albo w UI).

### Reuse z PRD #14 vs własna implementacja

Issues #15-20 (PRD #14) są zamrożone do czasu dostępności GC. Żeby ten Revolut feature działał teraz, **w tym issue budujemy minimalną wersję inbox + reguł + badge** — z myślą o tym że jak PRD #14 wskoczy, refactor zunifikuje dwa źródła.

To znaczy: tabele `transaction_inbox` + `categorization_rules` + kolumna `transactions.source` powstają **tutaj** (nie czekają na PRD #14). Pole `source` przyjmuje wartość `'revolut'` (analogicznie `'bank'` dla GC w przyszłości i `'manual'` dla domyślnych).

### Model danych (jednorazowo, używane też przez GC w przyszłości)

- `ALTER transactions ADD source TEXT DEFAULT 'manual'` ('manual' | 'revolut' | 'bank' w przyszłości)
- `ALTER transactions ADD external_inbox_id UUID NULL` — back-link do inbox row (nazwa generyczna, nie tylko bank)
- `transaction_inbox` — id, household_id, source ('revolut'), external_id UNIQUE (revolut transaction id), amount_enc, description_enc, booking_date, suggested_category, status, decided_at, created_at
- `categorization_rules` — id, household_id, pattern, category, created_by, hit_count, timestamps

Pola sensytywne (description, amount) szyfrowane `FINANCE_DATA_KEY`.

### Endpointy

- `GET /api/revolut/status` — admin-only, zwraca `{ connected: bool, last_synced_at, last_error? }`
- `POST /api/revolut/sync` — admin-only, manual trigger
- `GET /api/inbox?source=revolut&status=pending` — owner-only (admin jako owner)
- `POST /api/inbox/:id/approve { category, rememberRule? }`
- `POST /api/inbox/:id/dismiss`

### Filtry transakcji

- Skip transfery między własnymi rachunkami Revolut (po `type` z API)
- Mapowanie znaku: `amount > 0` → income, `< 0` → expense
- Multi-currency: Revolut może zwrócić w EUR, USD itp. — konwertuj na PLN po kursie z dnia transakcji (z API Revolut lub fallback na NBP) — to **TBD**, na początku obsługujemy tylko PLN i pomijamy inne walutami w pierwszej iteracji.

### Action log integration

Bulk wpis per sync: `action_log` row z `resource_type='external_import'`, `after={ source: 'revolut', count, accounts }`. Undo (24h okno) usuwa wszystkie wpisy z tego batcha.

### Setup admina

- Wejdź na https://business.revolut.com/ → Settings → APIs → Create new API certificate
- Wygeneruj client_id + JWT private key + initial authorization code
- One-time setup: POST do `/api/1.0/auth/token` z grant_type=authorization_code → dostajesz `refresh_token` (długoterminowy)
- `wrangler secret put REVOLUT_CLIENT_ID`
- `wrangler secret put REVOLUT_PRIVATE_KEY` (JWT signing key)
- `wrangler secret put REVOLUT_REFRESH_TOKEN`
- README sekcja "Setup admina Revolut sync" — krok po kroku ze screenshotami

## Out of Scope (tego issue)

- Multi-currency conversion — pierwsza iteracja tylko PLN, inne walutą skipowane z notyfikacją w inboxie
- Wsparcie dla innych member-ów gospodarstwa — tylko admin
- Wsparcie dla Revolut Personal (retail) — nie istnieje API
- Wsparcie dla GC/PKO/ING — to osobny tor (PRD #14, issues #15-20)
- Integracja z `savings_accounts` — bank-sync dla oszczędności poza scope

## Risks

1. **Revolut Business kosztuje €19/mies (Grow plan)** — to admin musi sam ponieść jako część konta firmowego. Jeśli nie ma JDG / nie chce płacić — feature się nie aktywuje.
2. **Token refresh sypanie się** — JWT-based auth Revolut jest delikatne, certyfikat ma czas życia. Mitigation: monitoring + email po 3 błędach.
3. **Multi-currency** — Revolut natywnie wspiera wiele walut. Pomijając je w pierwszej iteracji tracimy część transakcji. Akceptowalne dla MVP.
4. **PRD #14 może się rozjechać** — jak GC w końcu otworzy signupy i zaczniemy implementować PRD #14, możliwy rozjazd między `transaction_inbox` schema. Mitigation: pole `source` od początku przygotowane na multi-provider.

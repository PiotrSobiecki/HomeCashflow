## Type

**HITL** — wymaga manualnego dodania sekretów GC do Workers przez admina apki + rejestracji konta GoCardless.

## What to build

Faza fundamentalna ops/infra, widoczna tylko dla admina apki. Dodaje sekret `GOCARDLESS_SECRET_ID` i `GOCARDLESS_SECRET_KEY` do Cloudflare Workers (dev + prod), whitelistę email adminów (`ADMIN_EMAILS`), oraz cienki klient HTTP do GC z jedną tylko metodą potrzebną teraz — listowaniem aktywnych requisitions.

Wystawia endpoint `GET /api/admin/gc-usage` zwracający bieżące wykorzystanie konta GoCardless (X/50 + lista najwcześniej wygasających) — widoczny tylko dla emaili z whitelisty, dla innych 403.

Wprowadza helper `assertGcCapacity()` zwracający błąd `gc_limit_reached` gdy ≥50 aktywnych requisitions — Phase 2 użyje go w `POST /api/bank/connect` żeby hard-blokować nowe podpięcia po przekroczeniu limitu free tier.

Szczegóły implementacyjne w sekcji "Architectural decisions" i Phase 1 planu.

## Acceptance criteria

- [ ] `GOCARDLESS_SECRET_ID` i `GOCARDLESS_SECRET_KEY` ustawione przez `wrangler secret put` na dev i prod
- [ ] `ADMIN_EMAILS` (comma-separated) w env
- [ ] Cienki klient GC umie zrobić `GET /requisitions/?status=LN` z auth tokenem (test z mock fetch)
- [ ] `GET /api/admin/gc-usage` zwraca 200 z `{ used, limit: 50, expiring_soonest: [...] }` dla zalogowanego admina, 403 dla nie-admin-a
- [ ] Helper `assertGcCapacity()` zwraca błąd `gc_limit_reached` gdy >=50 — testowalny niezależnie
- [ ] Manual smoke test z REAL GC sandbox / staging account
